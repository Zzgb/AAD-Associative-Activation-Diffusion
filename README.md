# AAD — Associative Activation Diffusion

> 优化上下文窗口：用向量关联图谱替代平铺检索，LLM 沿关联逐跳探索，上下文按需加载不膨胀，关联结构防止遗忘。

## 动机：上下文窗口是稀缺资源

现代 LLM 支持 128K 甚至 1M Token 的上下文窗口。但窗口不是越大越免费——Token 消耗 = 成本 × 延迟，而且窗口后半段的注意力衰减严重，LLM 容易「忘记」靠前的内容。

当前 RAG 的做法是把 Top-K 检索结果**平铺进 Prompt**。查 GPU 返回 10 个片段，查 NVIDIA 又返回 10 个片段——全部堆在一起。片段之间没有关联标注，LLM 需要自己从一堆文本中推断「A 片段是 B 片段的上游供应商还是下游客户」。每次多轮追问，又要重新检索、重新塞入、重新推断。

## AAD 的思路：关联图谱 + 逐跳探索

AAD 把知识存成**节点 + 向量关联**的有向图。LLM 不一次性接收所有检索结果，而是像翻书一样逐页查阅：

```
用户: "黄仁勋的 GPU 公司有什么产品"

第 1 步 → aad_lookup("黄仁勋")
         ← 节点: 黄仁勋，关联: [创立了 NVIDIA] [领导 GPU 发明]
         上下文占用: ~200 Token（一个节点）

第 2 步 → aad_expand(NVIDIA 关联)
         ← 命中 NVIDIA，关联: [CUDA] [GeForce] [RTX] [数据中心]
         上下文占用: ~200 Token（一个节点）

第 3 步 → LLM: "信息够了，不用再查"
         → 输出答案
         总上下文占用: ~400 Token（两个节点 + 指令）
```

**关键区别**：RAG 一次塞入 10 个片段（可能 3000 Token），AAD 每步只加载当前节点（~200 Token），LLM 自主决定何时停止。窗口没满，注意力不散，推理链可追溯。

## 三个优化维度

**1. 窗口不膨胀。** 每步只加载一个节点的内容 + 关联概要。LLM 根据关联原因（如「创立了 NVIDIA」）判断下一步展开哪个方向，不需要把全部节点内容塞进窗口。

**2. 关联结构防止遗忘。** 传统 RAG 中检索片段是无序堆叠的，对话进行到第 5 轮时，第 1 轮的片段早就被挤出注意力范围。AAD 的节点之间通过向量关联显式绑定——GPU 的关联直接指向 NVIDIA，NVIDIA 指向 CUDA。即使某个节点的完整内容被压缩，关联结构仍然保留，LLM 能沿着「黄仁勋 → NVIDIA → CUDA」这条路径回溯。

**3. 规则节点。** 除了知识节点，AAD 图谱中可以嵌入**规则节点**——包含推理约束、格式要求、禁止事项等内容。LLM 在探索过程中命中规则节点时自动加载并遵守，无需每次手动注入 system prompt。规则和图谱数据共享同一套关联索引，LLM 在查阅数据的同时自然碰到对应的规则。

## 快速开始

```bash
# 1. 安装
python3 -m venv .venv && source .venv/bin/activate
pip install -e ".[dev]"

# 2. 配置
cp .env.example .env
# 编辑 .env，填入 DEEPSEEK_API_KEY

# 3. 运行
python -m aad.cli
# 首次运行自动生成 17 个种子节点，之后进入交互问答
```

## 架构

```
用户输入 → CLI → AADAgent.run()
  ├── 短期记忆 (SessionMemory) — 会话级，/quit 销毁
  │     ├── msg_N 消息节点（线性链）
  │     └── 长期记忆镜像（自动拷贝，内容可改写）
  │
  └── DeepSeek Chat API（工具调用循环，LLM 自主停止）
        ├── aad_lookup(name)           → 短期优先 → 长期 → 自动镜像
        ├── aad_expand(ref, top_k)     → 双索引搜索（短期 + 长期）
        └── aad_get_content(name)      → 获取节点完整内容
```

| 组件 | 说明 |
|------|------|
| **Agent LLM** | DeepSeek Chat API，工具调用原生支持 OpenAI function-calling 格式 |
| **嵌入** | 本地确定性 hash（SHA-256 → 归一化向量），零外部依赖 |
| **长期记忆** | `AADStore`：内存 HashMap + JSONL 持久化，跨会话保留 |
| **短期记忆** | `SessionMemory`：会话级知识图谱，自动镜像 + 推理关联，/quit 丢弃 |
| **向量索引** | FAISS `IndexFlatIP`（长短各一个独立索引） |
| **SDK** | `openai` Python SDK（`base_url` 指向 `api.deepseek.com/v1`） |

**图说明**：
- **双枢纽**：NVIDIA（7 条关联）和 GPU（8 条关联）是图中度最高的节点，承载大部分发散路径

- **长链**：黄仁勋 → NVIDIA → CUDA → 深度学习 → 人工智能，5 跳可达
- **交叉**：光线追踪同时被 图形渲染 和 RTX 指向，是两条路径的汇聚点
- **干扰项**：苹果/特斯拉/Python/量子计算 完全孤立，LLM 查询时应能识别「无关联」

## CLI 命令

| 命令 | 功能 |
|------|------|
| `> 你的问题` | 输入查询，Agent 探索图谱并回答 |
| `/nodes` | 列出所有长期记忆节点 |
| `/session` | 查看短期记忆状态（消息数、镜像节点） |
| `/quit`, `/q` | 退出（短期记忆自动清除） |
| `--reseed` | 启动时强制重生种子数据 |
